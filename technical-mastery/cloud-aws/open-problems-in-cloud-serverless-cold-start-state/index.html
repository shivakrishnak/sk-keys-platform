  3. Open Problems in Cloud Serverless (Cold Start, State)

⚡ TL;DR - Serverless on AWS (primarily AWS Lambda) is not solved technology - it has open engineering problems that affect production systems. The two central ones: (1) Cold Start: the first invocation of a Lambda function (or after it has been idle) requires initializing a new execution environment. For JVM-based Lambdas (Java, Kotlin, Scala): 1-5 seconds of initialization. For interpreted (Python, Node.js): 100-500ms. This latency is non-deterministic from the user’s perspective and can cause API timeouts for user-facing functions. Mitigations: SnapStart (Java): snapshots the initialized JVM after the first init (AWS creates a snapshot, future cold starts restore from snapshot: 90% latency reduction). Provisioned Concurrency: pre-warms N Lambda instances (always warm, never cold, but billed 24/7). (2) Serverless State: Lambda functions are stateless by design. No persistent in-process state survives between invocations. For stateful workflows: you need external state stores (DynamoDB, ElastiCache, S3). Step Functions provides serverless state machine orchestration (replaces hand-coded stateful workflows). The resulting problem: a simple task that would be one stateful service becomes a Lambda function + DynamoDB + Step Functions state machine + SQS queue + SNS + API Gateway = 6 services with 6 failure modes, 6 cost dimensions, and 6 observability surfaces. Serverless is genuinely simpler for event-driven, stateless, bursty workloads. It genuinely struggles for long-running, stateful, latency-sensitive workflows.

🔥 The Problem This Solves

The “serverless promises vs. reality” gap:

Serverless was marketed with these promises:

  1. “No infrastructure management”
  2. “Scales to zero cost when idle”
  3. “Scales to millions of requests automatically”
  4. “Pay per request, not per server”

All of these are true. The part that wasn’t in the marketing: serverless also introduces a class of problems that don’t exist with traditional server-based architectures.

The cold start problem in production:

A fintech startup migrates their REST API to API Gateway

  • Lambda (Java 11). Benchmarks in the dev environment: average response time 80ms. After deployment to production: P99 response time spikes to 3,500ms. Investigation: the P99 spikes correspond exactly to Lambda cold starts (JVM initialization). The business impact: 3.5 seconds P99 latency exceeds the payment processor’s 2-second timeout. Payments fail intermittently. The team rolls back to EC2 within 48 hours.

The state problem in production:

A data processing pipeline is designed as a Lambda function that processes messages from SQS. Requirements: maintain a “counter” of processed messages per customer (for rate limiting). First implementation: Lambda reads counter from DynamoDB, increments, writes back (conditional write for optimistic locking). Under load (5,000 messages/second): DynamoDB conditional write failures (optimistic lock contention). Second implementation: DynamoDB DAX cache. DAX helps reads but writes still contend. Third implementation: ElastiCache Redis with Lua script for atomic increment. What started as a 20-line stateful service became a Lambda + DynamoDB + DAX + ElastiCache + SQS + VPC deployment with $800/month infrastructure cost (vs. $50/month for a single ECS container with in-process state).

    - Configure network, storage, environment variables
    - Duration: 50-200ms (non-billed for first invocation)

  Phase 2: Function Init (customer-managed)
    - Execute code OUTSIDE the handler (module level)
    - Import libraries, create connections, load models
    - For Python/Node.js: 100-500ms typical
    - For JVM (Java/Kotlin): 1,000-5,000ms typical
    - For GraalVM native image: 50-200ms
    - This duration IS included in billable duration

  Phase 3: Handler execution
    - Execute the handler function
    - Return response

WARM INVOCATION (reuse of existing environment):
  Phase 3 only (handler execution)
  Phase 1 and Phase 2 skipped (environment already initialized)
  Latency: deterministic (only computation time)

Provisioned Concurrency:

Pre-initialized Lambda execution environments that are always-warm. Cost: billed per GB-second of provisioned concurrency (independent of whether they receive requests). Use case: latency-sensitive user-facing APIs. Not cost-effective for low-traffic functions (pay for idle capacity).

SnapStart (Java, GA November 2022):

After the initial Lambda cold start, AWS snapshots the state of the execution environment (JVM heap, stack, initialized libraries). Subsequent cold starts restore from the snapshot instead of re-initializing. Typical latency reduction: 90% (from 2,000ms to 200ms for a typical Spring Boot Lambda). Limitations: (1) no I/O during snapshot restore (connections must be re-established after restore - use @SnapshotHook), (2) requires Java 11+ on Corretto runtime, (3) snapshot state includes any secrets loaded at init time (security consideration).

Serverless state problem categories:

  1. In-function state between invocations: Lambda is stateless - any variable set in one invocation is NOT available in the next invocation (different execution environment). External state required (DynamoDB, ElastiCache).

  2. Long-running workflows: Lambda max timeout is 15 minutes. Multi-step workflows longer than 15 minutes cannot be expressed as a single Lambda. Step Functions required.

  3. Coordination between concurrent invocations: if multiple Lambda instances run simultaneously, they share no in-process state. Distributed coordination required (DynamoDB conditional writes, SQS, EventBridge).

🔩 First Principles Explanation

Why cold start is a fundamental property of serverless:

Serverless achieves “scales to zero” by completely stopping execution environments when there are no requests. When a request arrives: AWS must start a new execution environment. Starting an environment requires:

  1. Allocating compute resources (from a Firecracker MicroVM pool)
  2. Configuring the environment (VPC networking if VPC-attached, IAM, env vars)
  3. Initializing the runtime (Python interpreter, JVM, Node.js)
  4. Running customer initialization code (imports, class loading, DB connections)

Steps 1-3 are AWS-controlled and fast (50-200ms). Step 4 is customer-controlled and variable. JVM-based runtimes are slow to initialize because:

  • Class loading: JVM loads and JIT-compiles classes on first use
  • Spring Boot initialization: creates Spring Application Context (dependency injection framework), loads hundreds of beans
  • Connection pooling initialization: creates DB connection pools
  • Static initializers: customer code that runs at class load time

Why serverless statelessness is also fundamental:

Serverless achieves “scales horizontally” by running multiple independent instances with no coordination. Horizontal scaling works when instances are IDENTICAL and INTERCHANGEABLE. Instance A and Instance B can both handle any request.

For instances to be interchangeable: they must not rely on in-process state (state only one of them has). If Instance A accumulates a counter in memory: Instance B doesn’t have that counter. The request that goes to Instance B gets wrong state.

The solution: all state must live in a shared, external state store (DynamoDB, ElastiCache) that all Lambda instances can access. This is not a deficiency of serverless - it’s the trade-off that enables horizontal scaling. But it means: every stateful use case requires an external service with its own cost, latency, failure mode, and operational complexity.

Why SnapStart is not a complete solution:

SnapStart restores the JVM from a snapshot. The snapshot was taken at a specific point in time (after @PreSnapshot hook runs). When restored from snapshot, the JVM must:

  1. Re-establish any connections (database connections, HTTP clients) that were open at snapshot time (the connections are no longer valid after restore).
  2. Re-initialize any time-sensitive state (timestamps, random seeds, cached credentials with TTL).

SnapStart trades cold start latency for snapshot management complexity. It does NOT eliminate cold starts - it reduces their latency. A restored SnapStart environment still takes 200ms to become ready (vs. 2,000ms for a full JVM init).

import boto3
import json
from decimal import Decimal

def serverless_rate_limiter_with_dynamodb(
    customer_id: str,
    action: str,
    limit_per_minute: int = 100,
    region: str = 'us-east-1'
) -> dict:
    """
    Production pattern: Lambda-based rate limiter using DynamoDB.
    Lambda is stateless: this function has no in-process counter.
    State lives in DynamoDB (external, shared across all Lambda instances).

    DynamoDB atomic counter: UpdateItem with ADD (atomic increment)
    TTL: DynamoDB TTL deletes expired window entries automatically
    Concurrency: DynamoDB handles atomic increments without Lambda coordination
    """
    dynamodb = boto3.resource('dynamodb', region_name=region)
    table = dynamodb.Table('rate-limits')

    import time
    current_minute = int(time.time() / 60) * 60  # Floor to minute
    ttl = current_minute + 120  # 2-minute TTL (1 minute buffer)
    pk = f"{customer_id}#{action}#{current_minute}"

    try:
        # Atomic increment: ADD increments atomically across concurrent Lambda instances
        # ConditionExpression: fail if counter exceeds limit BEFORE increment
        # (optimistic check - prevents overshoot by 1 at max)
        response = table.update_item(
            Key={'pk': pk},
            UpdateExpression='ADD #cnt :one SET #ttl = :ttl',
            ConditionExpression=(
                'attribute_not_exists(#cnt) OR #cnt < :limit'
            ),
            ExpressionAttributeNames={
                '#cnt': 'count',
                '#ttl': 'ttl'
            },
            ExpressionAttributeValues={
                ':one': 1,
                ':limit': limit_per_minute,
                ':ttl': ttl
            },
            ReturnValues='UPDATED_NEW'
        )
        current_count = int(
            response['Attributes'].get('count', 1)
        )
        print(f"Allowed: {customer_id}/{action} count={current_count}/{limit_per_minute}")
        return {
            'allowed': True,
            'count': current_count,
            'limit': limit_per_minute
        }

    except dynamodb.meta.client.exceptions.ConditionalCheckFailedException:
        # Rate limit exceeded: condition check failed (count >= limit)
        print(f"RATE LIMITED: {customer_id}/{action} at {limit_per_minute}/min")
        return {
            'allowed': False,
            'count': limit_per_minute,
            'limit': limit_per_minute,
            'error': 'rate_limit_exceeded'
        }

Code walkthrough: The serverless_rate_limiter_with_dynamodb demonstrates the canonical external state pattern for Lambda. The ADD expression is atomic at DynamoDB level: even with 1000 concurrent Lambda instances calling this function simultaneously for the same pk, each increment is serialized by DynamoDB (no race conditions). The ConditionExpression prevents the count from exceeding the limit: if `count

= limit_per_minute, DynamoDB throws ConditionalCheckFailedException. The TTL (ttl` field) enables automatic cleanup: DynamoDB’s TTL process deletes items past their TTL without application intervention. This is the correct “serverless state for rate limiting” pattern. The downside: each Lambda invocation requires one DynamoDB write. At 10,000 requests/second: 10,000 DynamoDB writes/second. At $1.25/million writes: $0.0125/second = $1,080/day. For high-traffic rate limiting: consider Lambda@Edge + CloudFront Functions (which CAN maintain brief in-flight state via CloudFront distributed state) for cheaper per-request cost.

⚠️ Common Misconceptions

Misconception 1: “SnapStart eliminates cold starts for Java”

SnapStart reduces cold start LATENCY by ~90% but does NOT eliminate cold starts. With SnapStart: a cold start still occurs when a new Lambda execution environment is created (scale-out due to traffic burst, after extended idle period, after function deployment). The difference: instead of 1,500-5,000ms (JVM initialization), the cold start takes 150-200ms (snapshot restore). For user-facing APIs with a <200ms P99 requirement: SnapStart alone is insufficient. Provisioned Concurrency ELIMINATES cold starts (environments are always warm). The correct statement: “SnapStart makes Java Lambda cold starts fast enough for most user-facing APIs (200ms P99 is typically acceptable).”

Misconception 2: “Serverless is always cheaper”

Serverless pricing is based on actual usage. At HIGH SUSTAINED throughput, always-on compute (ECS Fargate, EC2) can be cheaper than Lambda. The key variables: average request duration and sustained request rate. Lambda is cost-optimal when: traffic is bursty (periods of zero or near-zero requests) and average request duration is short. At sustained high throughput (>1M requests/hour, 24/7): Lambda costs can exceed ECS Fargate. The decision is not “serverless is always cheaper” - it’s “use serverless for spiky/bursty workloads and always-on compute for sustained workloads.”

Misconception 3: “Step Functions replaces the need for external state”

Step Functions manages workflow state (which step is active, retry count, input/output of each step). It does NOT provide: shared application state between concurrent workflow executions, data caching between invocations, or database-like data storage. For application data (user records, processed results, accumulated counters): still need DynamoDB, ElastiCache, or other data stores. Step Functions solves the ORCHESTRATION state problem (tracking which Lambda to call next, handling retries/timeouts/errors in multi-step workflows). It does not solve the APPLICATION DATA state problem.

🚨 Failure Modes & Diagnosis

Failure 1: Lambda function running in VPC with slow cold starts (10+ seconds)

Symptom: Java Lambda in VPC takes 10-15 seconds to cold start. Same function without VPC: 2-3 seconds cold start. Adding VPC attached multiplied cold start latency by 5x.

Root cause: pre-2020, Lambda VPC functions required allocating a dedicated ENI (Elastic Network Interface) per execution environment at cold start time. ENI allocation from the EC2 ENI pool: 8-15 seconds. This was a known Lambda+VPC limitation.

Status: RESOLVED in 2019/2020 via Hyperplane ENIs. AWS pre-allocates ENI pools per VPC/subnet combination. VPC Lambda cold starts are now 1-2 seconds additional latency (not 10-15 seconds). If you’re seeing 10+ second VPC Lambda cold starts: your Lambda is likely deployed with the old ENI allocation mode (pre-Hyperplane).

Diagnosis:

# Check if function uses VPC:
aws lambda get-function-configuration \
  --function-name <function-name> \
  | python3 -c "import sys,json; c=json.load(sys.stdin); \
    print('VPC:', c.get('VpcConfig',{}).get('VpcId','none'))"

# Check Lambda ENI mode (Hyperplane vs. old):
# Check the account was enrolled in Hyperplane ENI improvement:
# All Lambda functions created AFTER September 2019: use Hyperplane
# Functions created BEFORE September 2019: may need manual update

# Fix: update Lambda function network config (triggers ENI reallocation):
aws lambda update-function-configuration \
  --function-name <function-name> \
  --vpc-config SubnetIds=<same-subnets>,SecurityGroupIds=<same-sgs>
# Updating VPC config triggers ENI migration to Hyperplane pool

Fix: if the function predates September 2019, update the VPC configuration (even with the same values). This triggers a re-enrollment in the Hyperplane ENI pool. Additionally: enable SnapStart (if Java) to reduce the remaining cold start latency after the ENI improvement.

Failure 2: Lambda reaching concurrency limit causing request rejections

Symptom: Lambda starts returning throttle errors (HTTP 429) under traffic load. CloudWatch shows Throttles metric increasing. API Gateway returns 429 to clients. Customer-visible error rate spike.

Root cause: Lambda has a per-region concurrency limit (default: 1,000 concurrent executions per account per region, soft limit adjustable to 10,000+). If all running Lambda functions across the region consume 1,000 concurrent executions simultaneously: new invocations are throttled.

Diagnosis:

aws lambda get-account-settings
# Look: ConcurrencyLimit (default: 1000)

  --namespace AWS/Lambda \

  --statistics Maximum \
  --period 60 \
  --start-time 2024-01-01T00:00:00Z \
  --end-time 2024-01-01T01:00:00Z

# Check throttle errors per function:

(2) Set Reserved Concurrency on critical functions to ensure they always have capacity (at the cost of limiting other functions). (3) Set maximum concurrency on batch processing functions to prevent them from consuming the full account limit. (4) Review whether Lambda is the right choice for sustained high-concurrency workloads (ECS/EKS may be more appropriate).

💎 Transferable Wisdom

Pattern: The “stateless scale-out” trade-off is universal in distributed systems

Lambda’s statelessness is not unique to serverless. The same trade-off appears at every layer of distributed systems:

  • HTTP servers (Nginx, Apache): stateless (session data in Redis, not in-process). Enables horizontal scaling.
  • Kubernetes pods: stateless by design. State in StatefulSets or external databases.
  • microservices: stateless (database is the state). Enables independent deployment and scaling.

In every case: moving state from in-process (fast, simple, single machine) to external (slightly slower, more complex, shared) enables horizontal scaling but introduces coordination overhead.

Lambda makes this trade-off EXPLICIT: you cannot accidentally write stateful Lambda code that works in production (it will break at scale when multiple instances run concurrently). EC2 instances and long-running processes CAN have in-process state - and this leads to subtle bugs when “horizontal scaling” means adding a second server that doesn’t share the first server’s in-process state.

Serverless forces the discipline of stateless-by-default. The complexity of external state (DynamoDB, ElastiCache) is the honest accounting of the true cost of horizontal scalability.

💡 The Surprising Truth

Lambda cold starts have been “almost solved” four times, and each solution introduced new constraints.

2014 (Lambda launch): “just keep functions warm” - call the function periodically (via CloudWatch scheduled event) to prevent cold starts. Works but wastes money and compute.

2018 (Provisioned Concurrency): reserved warm environments. Works, eliminates cold starts, but costs 24/7 regardless of traffic.

2019 (VPC Hyperplane ENI): eliminated the 10-15 second VPC cold start tax. Cold starts drop to 1-3 seconds for Java. “Java Lambda in VPC” went from unusable to merely problematic.

2022 (SnapStart for Java): restored JVM from snapshot. Cold starts drop to 150-200ms for Java. Works for most APIs. But: connections must be re-established on restore, snapshot security model requires review, only works for published versions.

The surprising part: after four major improvements, cold starts are still a topic of discussion. Why? The root cause (scaling to zero requires initializing from scratch) is a property of the serverless execution model itself. Each solution is an engineering workaround for the fundamental trade-off, not an elimination of it. SnapStart is the best workaround yet for Java. The only complete solution: don’t scale to zero (Provisioned Concurrency - but then you’re not fully benefiting from the serverless cost model).

The meta-lesson: some architectural trade-offs cannot be fully engineered away. They can be managed (mitigated), but not eliminated without sacrificing the core property that creates them.

🧠 Think About This Before We Continue

  1. A Lambda function uses Python 3.12 and makes a connection to an RDS PostgreSQL database at module level (outside the handler). The connection is made during cold start (module-level import). Is this a good practice? (Answer: it depends. Making the connection at module level (not inside the handler) is intentional “connection caching”: the connection is made once per Lambda execution environment and reused across multiple invocations of the same warm environment. This IS a best practice: reduces connection establishment overhead on each invocation. However: the connection WILL be broken eventually (idle timeout, RDS proxy recycles connections, Lambda environment recycled). The handler must handle reconnection gracefully (catch OperationalError, reconnect). The pattern: check connection is alive before query (conn.closed == 0), reconnect if not. With RDS Proxy: the proxy keeps connections alive and multiplexes Lambda connections, making this pattern more reliable. Without RDS Proxy: at scale (1,000 concurrent Lambda invocations = 1,000 concurrent DB connections), you’ll exhaust RDS max_connections. RDS Proxy is strongly recommended for Lambda+RDS architectures.)

  2. You’re designing a real-time chat application where users send messages that must be delivered to online recipients within 100ms. Would you use Lambda for message routing? Why or why not? (Answer: No, not Lambda alone. Two problems: (1) cold start: Lambda can take 100ms-5s to cold start, which violates the 100ms delivery requirement. (2) WebSocket connections: Lambda functions are stateless and short-lived. WebSocket connections are long-lived (minutes to hours). Lambda cannot maintain a persistent WebSocket connection per user. The correct architecture: API Gateway WebSocket API (maintains WebSocket connections, manages connection IDs) + Lambda for message routing logic (short-lived, handles one message at a time) + connection store in DynamoDB (which connection IDs are online). The Lambda handles message routing by looking up the recipient’s connection ID in DynamoDB, then calling API Gateway Management API to push the message to that connection. Cold start mitigation: Provisioned Concurrency on the message routing Lambda (100ms delivery requirement cannot tolerate cold starts).)

  3. SnapStart snapshots the JVM state after initialization. The initialization code loaded AWS Secrets Manager credentials into a class-level variable. What security concern does this create? (Answer: The credentials are included in the SnapStart snapshot. The snapshot is stored by AWS (encrypted), but: (1) if the credentials are rotated (AWS Secrets Manager rotation) AFTER the snapshot is created: the snapshot contains the OLD credentials. Lambda instances restored from the old snapshot will use the old credentials until the snapshot is refreshed. This can cause: intermittent authentication failures (some invocations use old credentials, some are fresh). Fix: use @PostRestore hook to reload credentials from Secrets Manager AFTER restore (not from the cached snapshot value). (2) Access to the snapshot = access to the credentials at snapshot time. AWS manages snapshot encryption and access control, but this is an additional attack surface. Best practice: do NOT cache secrets in static/module-level variables for SnapStart functions. Always refresh secrets in @PostRestore or in the handler with appropriate caching (e.g., re-fetch if >5 minutes old).)

🗺️ Model Answer Framework

WHAT IT IS: Open serverless problems: (1) Cold start - Lambda execution environment initialization latency (JVM: 1-5s, Python: 100-500ms). Mitigations: SnapStart (Java, 150-200ms), Provisioned Concurrency (zero cold start, 24/7 cost). (2) Statelessness - no in-process state between invocations. Requires external state (DynamoDB, ElastiCache, Step Functions for workflows).

WHY IT MATTERS: Cold start causes P99 latency spikes for user-facing APIs. Statelessness creates coordination overhead that can exceed the cost savings of serverless for complex stateful workloads. Understanding these problems = knowing when serverless is the right tool and when it isn’t.

HOW IT WORKS: Cold start phases: AWS init (Firecracker MicroVM, ~50-150ms) + customer init (library imports, JVM class loading, ~100ms-5s). SnapStart: CRaC snapshot of JVM heap/stack post-init, restore is 10x faster than re-init. Step Functions: manages workflow state (which step, retries, timeouts) externally to Lambda.

TRADE-OFFS:

  • SnapStart vs. Provisioned Concurrency: cost vs. cold start elimination
  • Lambda vs. ECS: serverless simplicity vs. sustained throughput cost
  • In-process state (fast) vs. external state (scalable): always external for Lambda
  • Lambda@Edge vs. CloudFront Functions: flexibility vs. cost vs. latency

import com.amazonaws.services.lambda.crac.*;

    @PreSnapshot

        // Close all connections before snapshot
        DatabaseConnectionPool.close();
        HttpClientPool.close();

    @PostRestore
    public static void afterRestore() {
        // Re-establish connections after snapshot restore
        DatabaseConnectionPool.initialize();
        HttpClientPool.initialize();

}
