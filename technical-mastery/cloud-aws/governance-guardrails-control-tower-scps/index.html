⚡ TL;DR - AWS Governance is the combination of preventive controls (Service Control Policies block actions before they happen), detective controls (AWS Config rules detect policy drift after it happens), and corrective controls (Lambda or SSM Automation remediates violations). AWS Control Tower automates this governance framework across a multi-account organization: it deploys SCPs, Config rules, CloudTrail aggregation, and IAM Identity Center in a repeatable landing zone. The most critical SCP: deny s3:DeleteBucketPolicy and cloudtrail:StopLogging to prevent security coverage gaps. The most common governance failure: SCPs exist in management account but development accounts are in the root OU (no SCPs applied). Account isolation is only as strong as the OU structure.

🔥 The Problem This Solves

“Our organization has 50 AWS accounts managed by 20 development teams. Last month: Team A accidentally made an S3 bucket public (data exposure). Team B turned off CloudTrail in their dev account (audit gap). Team C launched a g4dn.24xlarge GPU instance in an account with no budget limit ($50K/month uncapped cost). Team D enabled an AWS service in eu-east-1 (data residency violation - our compliance requires EU-West only). All four incidents were preventable. Three were detected only at month-end billing review or security audit. The fourth (S3 public bucket) was detected by an external researcher. We have 50 accounts, 200+ engineers, and no automated guardrails. How do we prevent the next incident?”

This is the governance gap problem: multi-account AWS environments without automated guardrails produce a class of preventable incidents that manual processes (code review, security audits) cannot reliably catch. Governance in AWS must be automated at the account level, not enforced through human review.

📘 Textbook Definition

AWS governance is the set of preventive, detective, and corrective controls applied across an AWS Organization to enforce security policies, compliance requirements, and operational standards at scale. AWS provides three primary governance mechanisms:

Service Control Policies (SCPs): IAM-like policies attached to AWS Organizations OUs that restrict actions available to accounts within the OU. SCPs override any account-level IAM policy: even an IAM Admin user cannot perform an action that an SCP denies. They are additive with deny-list (permissive) or allow-list (restrictive) strategies.

AWS Config Rules: Continuously evaluate AWS resource configurations against compliance rules. Detective controls: they detect policy drift after resources are created or modified. Config supports managed rules (pre-built by AWS) and custom rules (Lambda functions for complex logic).

AWS Control Tower: Managed service that orchestrates multi-account governance. Control Tower provisions the landing zone (Account Factory, OU structure, log archive account, audit account), deploys Control Tower Guardrails (built-in SCPs + Config rules), and detects drift from baseline.

⏱️ Understand It in 30 Seconds

One line: AWS governance prevents security and compliance violations across all accounts using three control types: SCPs (prevent the action), Config rules (detect the violation), and remediation Lambda (fix the violation) - enforced uniformly across accounts via AWS Organizations.

One analogy:

AWS governance is like a building’s fire safety system: fire doors (preventive controls - SCPs) block fire from spreading to other floors. Smoke detectors (detective controls - Config rules) alert when a violation occurs. Sprinklers (corrective controls - automated remediation) activate to fix the problem. A building without fire doors relies on occupants not starting fires. A building without smoke detectors discovers fires late. A well-governed AWS organization has all three: actions that violate policy are blocked (fire doors), policy drift is detected within minutes (smoke detectors), and violations are auto-remediated where possible (sprinklers). The goal is not to prevent all human mistakes but to make human mistakes recoverable (detected and remediated quickly).

One insight: The most important SCP is not the one that prevents security violations - it’s the SCP that prevents governance from being turned off: Deny cloudtrail:StopLogging, Deny config:DeleteConfigurationRecorder, Deny organizations:LeaveOrganization. These protect the detective control infrastructure itself. An attacker or accidental mistake that turns off CloudTrail can operate in an account for weeks without leaving audit trails. Protecting the security logging infrastructure is the meta-governance requirement.

Persona pitches:

Junior Developer: When you create an AWS resource in a governed account, your actions are constrained by SCPs. “Access Denied” when creating a resource in an unauthorized region is an SCP in action. AWS Config checks whether your resource is compliant (e.g., S3 bucket is not public, RDS instance is encrypted). You may receive a Config notification when your resource is non-compliant. These are organizational guardrails, not permission bugs.

Mid-Level Architect: The SCP decision framework: deny-list vs. allow-list strategy. Deny-list (add deny rules for specific actions you want to prevent) is less restrictive but harder to reason about completeness. Allow-list (deny all actions, then allow specific services/actions explicitly) is more restrictive but provides complete coverage. For most organizations: deny-list for development accounts (freedom with guardrails), allow-list for production accounts (strict: only explicitly approved services).

Skeptical Reviewer: “SCPs are difficult to test and can break production if misconfigured.” Valid. SCP misconfiguration is a real risk: an SCP that accidentally denies sts:AssumeRole will break all CI/CD pipelines that assume IAM roles. Mitigation: (1) Test SCPs in a non-production OU first, (2) Use AWS Organizations Policy Simulator to test before deployment, (3) SCP changes go through code review (not console), (4) Never apply SCP to the management account (recovery path).

🧠 Mental Model / Analogy

SCP vs. IAM policy is like national law vs. local company policy. A national law (SCP) applies to everyone in the country regardless of which company they work for (account). A company’s HR policy (IAM policy) applies within the company. A company cannot create an HR policy that violates national law. If national law says “companies cannot employ workers under 16,” no HR policy can override that. Similarly, if an SCP says “deny s3:DeleteBucket,” no IAM Admin policy within the account can override it. IAM policies operate within the space allowed by SCPs. Understanding this hierarchy is critical: if an SCP blocks an action, troubleshooting IAM policies in the account is pointless - fix the SCP.

📶 Gradual Depth - Five Levels

Level 1 (5-year-old): AWS guardrails are rules that everyone in the company must follow when using AWS. Some rules stop bad things from happening. Other rules check that everything is set up correctly. If something is wrong, the rules send an alert.

Level 2 (Junior Developer): SCPs restrict what IAM users in an AWS account can do. Even an IAM Admin can’t do something that an SCP blocks. Config rules check whether your resources meet security and compliance standards (e.g., S3 buckets must not be public). If a rule is violated, you may see an alert or the resource may be auto-fixed.

Level 3 (Mid Engineer): Three control types: preventive (SCP blocks the action), detective (Config detects the drift), corrective (Lambda auto-remediates). Control Tower automates these across a multi-account organization. Key SCPs to always deploy: deny CloudTrail StopLogging, deny Config DeleteConfigurationRecorder (protect the audit infrastructure itself), deny LeaveOrganization, deny region-specific service calls if data residency is required.

Level 4 (Senior Engineer): SCP strategy considerations: deny-list provides flexibility (teams can use any service, just not the denied actions). Allow-list is strict (explicit permit for each service). In practice: most organizations use deny-list in non-production OUs. Allow-list is reserved for heavily regulated production accounts (PCI DSS, HIPAA). Config Conformance Packs provide pre-built rule collections for common compliance frameworks - faster than building individual Config rules for each requirement. Config Aggregator provides org-wide visibility: “Which accounts have the most non-compliant resources?”

Level 5 (Principal/Architect): SCPs and governance-as-code: all SCPs should be defined in Terraform (or CDK) and deployed via CI/CD pipeline. This ensures: version control (can revert bad SCP change), code review (second pair of eyes before deployment), audit trail (who changed what SCP when), and testability (pre-deployment test suite). The Control Tower Customizations (CfCT) or Landing Zone Accelerator on AWS provides this GitOps governance model for large organizations. Detective governance gap: Config rules evaluate resources at creation and on change, but NOT historically. A resource created before a Config rule was deployed is not evaluated until it changes. Initial compliance assessment: Config remediation triggers periodic evaluation (or Lambda to trigger evaluation for all existing resources when a new rule is deployed).

Scenario 2: AWS Config custom rule for tagging compliance

# Lambda function for AWS Config custom rule
# Enforces required tags on EC2 instances
import boto3
import json
from datetime import datetime

REQUIRED_TAGS = {
    'Team': None,         # Any non-empty value
    'Environment': ['production', 'staging', 'development'],
    'CostCenter': None    # Any non-empty value
}

def lambda_handler(event, context):
    """
    AWS Config custom rule evaluation:
    EC2 instances must have required tags with valid values.
    """
    invoking_event = json.loads(event['invokingEvent'])
    configuration_item = invoking_event.get('configurationItem', {})
    
    # Only evaluate EC2 instances
    if configuration_item.get('resourceType') != 'AWS::EC2::Instance':
        return put_evaluation(event, 'NOT_APPLICABLE', configuration_item)
    
    # Skip terminated instances
    if configuration_item.get('configurationItemStatus') == 'ResourceDeleted':
        return put_evaluation(event, 'NOT_APPLICABLE', configuration_item)
    
    tags = configuration_item.get('tags', {})
    
    violations = []
    for tag_key, allowed_values in REQUIRED_TAGS.items():
        tag_value = tags.get(tag_key)
        
        if tag_value is None or tag_value.strip() == '':
            violations.append(f"Missing tag: {tag_key}")
            continue
        
        if allowed_values and tag_value.lower() not in allowed_values:
            violations.append(
                f"Tag {tag_key}='{tag_value}' not in "
                f"allowed values: {allowed_values}"
            )
    
    if violations:
        annotation = f"Tag violations: {'; '.join(violations)}"
        compliance = 'NON_COMPLIANT'
    else:
        annotation = 'All required tags present and valid'
        compliance = 'COMPLIANT'
    
    return put_evaluation(event, compliance, configuration_item, annotation)

def put_evaluation(event, compliance, config_item, annotation=''):
    config = boto3.client('config')
    config.put_evaluations(
        Evaluations=[{
            'ComplianceResourceType': config_item['resourceType'],
            'ComplianceResourceId': config_item['resourceId'],
            'ComplianceType': compliance,
            'Annotation': annotation[:256],  # Max 256 chars
            'OrderingTimestamp': datetime.strptime(
                config_item['configurationItemCaptureTime'],
                '%Y-%m-%dT%H:%M:%S.%fZ'
            )
        }],
        ResultToken=event['resultToken']
    )

# Result: 40% of EC2 instances untagged in production.
# Cost allocation impossible. Security audit failures.

# GOOD: Config custom rule + auto-notification
# NON_COMPLIANT triggers SNS to team lead.
# After 48 hours without tagging: SSM Automation

Code walkthrough: The Config custom rule demonstrates the pattern for organizational tag enforcement. The REQUIRED_TAGS dictionary with allowed_values provides both presence validation (Team must exist) and value validation (Environment must be ‘production’, ‘staging’, or ‘development’ - no free-form values). The annotation[:256] truncation is critical: Config evaluation annotations have a 256-character limit; exceeding it causes the Lambda invocation to fail with an unrelated error. The deletion check (ResourceDeleted status) prevents violations being reported for terminated instances. The BAD/GOOD pattern: relying on teams to self-tag (BAD) has 40% non-compliance in most organizations. Automated enforcement with escalation (GOOD) drives tagging compliance to > 95% within 90 days.

# Lambda for Config auto-remediation:
# Automatically block public access on non-compliant S3 buckets

import boto3
import json
import os

def lambda_handler(event, context):
    """
    Auto-remediation for s3-bucket-public-read-prohibited.
    Called by Config remediation action when bucket is NON_COMPLIANT.
    
    Input: Config remediation execution event with bucket name.
    """
    # Config remediation passes resource ID as parameter
    resource_id = event.get('resourceId') or \
                  event['executionControls']['SsmControls'].get('resource')
    
    # Extract bucket name from resource ID
    # Config resource ID for S3: just the bucket name
    bucket_name = resource_id
    
    s3 = boto3.client('s3')
    sns = boto3.client('sns')
    
    # Block all public access
    try:
        s3.put_public_access_block(
            Bucket=bucket_name,
            PublicAccessBlockConfiguration={
                'BlockPublicAcls': True,
                'IgnorePublicAcls': True,
                'BlockPublicPolicy': True,
                'RestrictPublicBuckets': True
            }
        )
        
        remediation_status = 'REMEDIATED'
        message = (
            f"AUTO-REMEDIATED: S3 bucket '{bucket_name}' was made "
            f"public. Public access has been blocked automatically.\n\n"
            f"ACTION REQUIRED: Review why this bucket was made public. "
            f"If public access is required, follow the exception process "
            f"at: confluence.internal/aws-governance/s3-public-exception\n\n"
            f"CloudTrail investigation: Filter by "
            f"eventName=PutBucketAcl OR PutBucketPublicAccessBlock "
            f"for bucket {bucket_name} in the last 24 hours to identify "
            f"the IAM principal that made the bucket public."
        )
    except Exception as e:
        remediation_status = 'FAILED'
        message = (
            f"AUTO-REMEDIATION FAILED: S3 bucket '{bucket_name}' "
            f"is still public. Manual intervention required.\n"
            f"Error: {str(e)}"
        )
    
    # Notify security team
    topic_arn = os.environ['SECURITY_ALERTS_SNS_TOPIC_ARN']
    sns.publish(
        TopicArn=topic_arn,
        Subject=f"[{remediation_status}] Public S3 Bucket: {bucket_name}",
        Message=message
    )
    
    return {
        'status': remediation_status,
        'bucket': bucket_name
    }

⚠️ Common Misconceptions

Misconception 1: “SCPs apply to the management account” Reality: SCPs explicitly do NOT apply to the management account (formerly called the master account). The management account is the root of the organization

  • it’s the emergency recovery path if SCPs block access. Governance teams should use the management account ONLY for organization management tasks, not for workloads. Applying SCPs to the management account is technically not possible and by design: you always have an escape hatch.

Misconception 2: “SCPs grant permissions” Reality: SCPs only restrict permissions - they never grant them. A principal still needs explicit Allow in an IAM policy to perform an action. SCPs define the maximum permission boundary (what CAN be granted by IAM policies). An SCP that allows s3:* does not mean all IAM principals can access S3 - they still need IAM policies that grant S3 access. SCP = maximum possible permissions; IAM policy = actual granted permissions (within SCP boundaries).

Misconception 3: “Control Tower manages all SCPs and Config rules” Reality: Control Tower manages SPECIFIC guardrails (called Control Tower Guardrails/Controls). These are a subset of possible governance controls. Organizations with specific compliance requirements need ADDITIONAL SCPs and Config rules beyond what Control Tower provides. Control Tower Customizations for Control Tower (CfCT) is the mechanism for deploying custom SCPs and Config rules alongside Control Tower’s built-in guardrails. Control Tower ≠ complete governance; it’s the foundation.

Misconception 4: “Config evaluates resources before they’re created (preventive)” Reality: AWS Config is a detective control, not preventive. It evaluates resources AFTER they are created or modified. A Config rule cannot prevent an EC2 instance from being launched in the wrong region - it can only detect it and trigger remediation afterward. Preventive controls require SCPs. The typical misunderstanding leads to governance designs with only Config rules, which leaves the window of violation open (between creation and detection, which can be minutes to hours).

Misconception 5: “CloudFormation StackSets are sufficient for multi-account Config deployment” Reality: CloudFormation StackSets with SERVICE_MANAGED deployment deploy Config rules to all accounts in an OU, but they don’t handle: (1) new accounts automatically getting Config rules (requires StackSet automatic deployment enabled), (2) drift detection (StackSets can drift), (3) accounts that leave and rejoin the OU. Control Tower with AWS Config (delegated administrator from the audit account) provides better operational guarantees for multi-account Config governance.

🚨 Failure Modes & Diagnosis

Failure Mode 1: SCP breaks CI/CD pipelines

SYMPTOM:
  All CI/CD pipelines in an OU fail immediately
  after an SCP change. Error: "AccessDenied:
  User is not authorized to perform sts:AssumeRole"

  New SCP in the production OU inadvertently denies
  sts:AssumeRole. CI/CD pipeline workers (EC2,
  Lambda, or ECS) assume IAM roles in member accounts.
  SCP blocks the role assumption.
  
  Common cause: SCP author added
  "Deny sts:AssumeRole unless MFA present"
  (intended for human access)
  but did not add a condition to exempt CI/CD
  service principals (machine-to-machine role assumption
  cannot use MFA).
  
DIAGNOSIS:
  CloudTrail: search for recent "AccessDenied"
  events with eventName=AssumeRole, errorCode=
  "AccessDenied". Note the userIdentity.principalId.
  
  AWS Organizations Policy Simulator:
  Test AssumeRole against the new SCP for the
  CI/CD service role. Verify the SCP is the cause.
  
FIX:
  Add condition to the SCP:
  "AssumeRole denied only if principal is a human user,
  not a service role."
  Condition:
    StringNotLike:
      aws:PrincipalArn:
        - "arn:aws:iam::*:role/cicd-*"
        - "arn:aws:iam::*:role/automation-*"
  
  PREVENTIVE MEASURE:
  1. All SCPs tested in sandbox OU before production.
  2. Post-deployment: 30-minute CloudTrail monitoring
     for unexpected AccessDenied events.
  3. SCP changes: separate PR, mandatory second reviewer.

💎 Transferable Wisdom

Pattern: Policy-as-Code for Governance SCPs and Config rules as code (Terraform, CDK) follows the same principle as Infrastructure as Code: governance policies should have version control, code review, and deployment pipelines. The alternative (console-driven SCP management) creates undocumented, untested policy changes that cannot be audited.

  • Network firewall rules as code: Firewall rules (ACLs, security group rules, WAF rules) have the same properties as SCPs: they should be version-controlled, code-reviewed, tested, and deployed via pipeline. Security teams that manage firewall rules in the console have the same audit and drift problems as cloud teams that manage SCPs in the console.
  • Database access policies: Database-level access controls (GRANT/REVOKE in Postgres, MySQL) are governance for data access. Managing them as code (Terraform PostgreSQL provider, Vault DB secrets engine) applies the same governance principles.
  • Kubernetes RBAC/OPA (Open Policy Agent): OPA Gatekeeper is Kubernetes’ equivalent of SCPs: admission webhooks that block resource creation if it violates policy. Same three control types: OPA (preventive), Falco (detective), operator (corrective).

Universal principle: In any system that grants users capabilities, capability abuse is proportional to the gap between “what users can do” and “what users should do.” Governance closes that gap at three levels: prevent (reduce the gap by limiting capabilities), detect (alert when the gap is exceeded), and correct (restore the correct state when violated). Governance is most effective when it’s automated (can’t be bypassed by intent or accident) rather than procedural (relies on human adherence).

💡 The Surprising Truth

AWS SCPs have a counterintuitive property: they apply to the ROOT USER of each member account. This means an SCP can deny actions that the root user (normally omnipotent in an AWS account) cannot perform. This is the only mechanism in AWS to restrict the root user’s capabilities. It’s also why SCPs should NEVER be applied to the management account: if an SCP accidentally blocks the management account root user’s ability to manage the organization, recovery requires AWS Support intervention.

The production implication: organizations that have previously relied on “emergency root user access will always work” as a fallback for account-level incidents need to re-evaluate this assumption in SCPs-governed accounts. The BreakGlass role pattern (IAM role with MFA + limited exemption from SCPs) is a safer emergency access mechanism than root user access, because it provides audit trails, requires MFA, and is scope-limited. Root user emergency access in an SCP-governed account may be blocked - the BreakGlass role is the correct design.

  1. A developer tries to assume an IAM role in a production account and receives “AccessDenied.” The IAM policy for the role explicitly allows sts:AssumeRole, and the trust policy includes the developer’s role as a trusted principal. The organization has SCPs. How do you systematically diagnose this?

    Hint: When IAM policies are correctly configured but access is denied: the cause is above IAM (permission boundary, SCP, or resource policy). Step 1: AWS Organizations Policy Simulator - test the specific action against the OU’s SCP hierarchy. If the simulator shows a Deny: the SCP is the cause. Step 2: Check if the production OU has an SCP that denies sts:AssumeRole without an exemption for this specific role. Step 3: Identify which SCP statement is denying - the Sid helps. Fix: add the role to the exemption condition in the SCP. The diagnostic heuristic: “IAM policy correct + access denied = look above IAM.” SCPs, permission boundaries, and resource policies are the candidates. CloudTrail errorCode “AccessDenied” with errorMessage “explicit deny in a service control policy” confirms SCP is the cause.

  2. You deploy a new Config rule that requires all EC2 instances to have IMDSv2 enforced (HttpTokens=required). The rule immediately shows 1,200 NON_COMPLIANT instances. Auto-remediation would update all 1,200 instances to force IMDSv2. Should you enable auto-remediation?

    Hint: Auto-remediation for IMDSv2 enforcement is RISKY. Changing HttpTokens from ‘optional’ to ‘required’ on a running EC2 instance means that any application code that calls IMDSv1 (without token in the request headers) will stop receiving metadata responses. This breaks legacy application code that uses IMDSv1 for credential retrieval (e.g., old AWS SDK versions). Auto-remediating 1,200 production instances simultaneously could cause widespread application failures. Correct approach: (1) Identify which instances run applications that use IMDS (CloudWatch Metric: MetadataNoToken count > 0 = still using IMDSv1). (2) For instances with zero IMDSv1 calls (safe): auto-remediate. (3) For instances with IMDSv1 calls: manual remediation - update application code first, then enforce IMDSv2. Rule of thumb: auto-remediate only when the remediation is safe for ALL instances in scope.

  3. Your organization has 100 AWS accounts. The security team wants to deploy a new baseline SCP to all accounts immediately. The SCP denies creation of IAM users with console access passwords. What is the correct deployment process?

    Hint: The correct deployment process treats SCP changes like production code changes, not configuration changes. (1) Test in sandbox OU first (1-2 accounts). Create an IAM user with a console password - verify “AccessDenied.” Verify existing operations still work. (2) Check impact: which accounts have existing IAM users with console passwords? AWS Config rule (iam-no-inline-policy-check or custom Lambda) can identify them. These accounts need remediation (migrate to IAM Identity Center) before the SCP is applied or their console IAM users will still exist but no new ones can be created. (3) Apply to non-production OUs. 24-hour observation. (4) Apply to production OUs. Monitor CloudTrail for unexpected Access Denied events for 30 minutes post-deployment. The principle: no SCP is deployed to production without a staged rollout and verification. A broken SCP affecting 100 accounts can paralyze an organization.

🏗️ Model Answer Framework

WHAT is AWS governance (SCPs, Config, Control Tower)? AWS governance is the combination of preventive controls (SCPs block prohibited actions before they happen), detective controls (Config rules detect policy drift after resources are created), and corrective controls (auto-remediation Lambda fixes violations automatically). AWS Control Tower orchestrates this framework across a multi-account organization with built-in guardrails.

WHY do governance controls matter at organizational scale? With 50+ accounts and 200+ engineers, human review of every configuration change is not feasible. Automated governance scales with account count: an SCP applied to an OU governs 50 accounts with zero additional overhead vs. 1 account. Manual governance does not scale: each new account or developer increases the review burden linearly.

HOW do you design a governance framework? Three steps: (1) Identify compliance requirements (data residency, audit logging, encryption requirements). (2) Map each requirement to the appropriate control type (prevent with SCP vs. detect with Config vs. correct with Lambda). (3) Deploy in stages (sandbox → development OU → production OU), verify after each stage. Always test in non-production first: a misconfigured SCP that blocks critical actions in production can cause widespread outages.

  • Custom Config rules: required tags (Team, Environment, CostCenter), CloudTrail enabled.

  • Deny all other services.

  • CloudWatch alert for any PCI account non-compliance.

Q2: How do SCPs differ from IAM permission boundaries, and when do you use each?

Expert answer: “Both restrict permissions but at different levels and for different purposes.

SCPs operate at the AWS Organizations level: they restrict what ALL principals in an account can do. They’re set by the organization administrator and cannot be modified by account administrators. SCPs are organizational controls.

Permission boundaries operate at the IAM entity level: they restrict what a specific IAM role or user can do, even if their IAM policies allow more. They’re set by an IAM administrator within an account (with iam:PutUserPermissionsBoundary or iam:PutRolePermissionsBoundary).

Use SCPs when: You want organization-wide enforcement (e.g., no one in any production account can disable CloudTrail). The control should not be modifiable by account administrators. You need to enforce compliance requirements across all accounts uniformly (data residency, audit logging).

Use permission boundaries when: You want to delegate IAM administration to development teams while constraining what IAM roles they can create. Example: ‘Developer teams can create IAM roles for their Lambda functions, but all roles created by developers must have a permission boundary that limits them to S3, DynamoDB, and CloudWatch - they cannot create roles with admin access.’

The distinction: SCPs apply to ALL principals in the account (account-level ceiling). Permission boundaries apply to SPECIFIC IAM entities (entity-level ceiling). In combination: SCP sets the account ceiling (no one can use services outside eu-west-1), and permission boundary sets the entity ceiling within that account (this Lambda role can only access DynamoDB, not EC2, even though EC2 is allowed by the SCP).”
