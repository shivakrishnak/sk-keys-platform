⚡ TL;DR - Buck2 is Meta’s open-source, Rust-based build system that replaces Buck1 with a fundamentally redesigned analysis and execution model. Like Bazel, it uses Starlark BUILD files and supports remote execution via REAPI. Unlike Bazel, Buck2 integrates the build graph incrementally at all phases - analysis, execution, and testing are unified in a single daemon process. The result is dramatically faster incremental builds and more responsive developer feedback. Buck2 also introduces bxl (Buck2 Extension Language) for programmatic build graph queries, and integrates with Meta’s tpx test runner for unified test execution. Core design principle: every computation in the build graph is memoized; re-running the same query twice always returns instantly.

🔥 The Problem This Solves

The Limitations of Buck1 and Bazel at Meta’s Scale:

Meta’s monorepo contains hundreds of millions of lines of code. At that scale, even Bazel’s incremental build model showed performance problems:

  1. Eager re-analysis on any BUILD file change. In Buck1 (and partially in early Bazel), changing a BUILD file could invalidate large portions of the build graph, triggering expensive re-analysis.

  2. Separate daemon processes for analysis vs execution. Build systems traditionally separate “what to build” (analysis) from “how to build it” (execution). At Meta’s scale, coordinating these phases via IPC added latency.

  3. Limited incrementality for test results. Re-running tests after a code change required re-executing all potentially-affected tests even if the test binary itself was unchanged and its inputs were identical.

  4. Buck1 was JVM-based. Startup time and GC pauses were visible pain points for developer iteration speed.

What Buck2 does differently:

Buck2 is written in Rust (low latency, no GC pauses, fast startup). Its core innovation is a unified incremental computation framework called DICE (Distributed Incremental Computation Engine). Every computation in the build graph - loading BUILD files, running rules, executing actions, evaluating tests - is a DICE node. Any re-query of an unchanged DICE node returns its cached result immediately, without re-computation. This makes buck2 build after a comment- only change return in milliseconds.

📘 Textbook Definition

Buck2 is an open-source build system developed by Meta, released publicly in April 2023. It is written in Rust and designed as a successor to Buck (retroactively called Buck1). Key characteristics:

  • Starlark BUILD files: Compatible with Bazel’s BUILD file syntax (both use Starlark, the dialect of Python). Existing .bzl rule libraries can often be ported to Buck2 with minor changes.

  • DICE (Distributed Incremental Computation Engine): Buck2’s internal computation model treats all build graph nodes as memoized function calls. Stale nodes are recomputed lazily; clean nodes return cached results. This unifies what were separate phases in Bazel (loading/analysis/execution) into one consistent incremental framework.

  • bxl (Buck2 Extension Language): A Starlark-based scripting API that allows querying and manipulating the build graph programmatically. bxl scripts can query the dependency graph, inspect action graphs, and generate outputs - similar to bazel query/cquery/aquery but with scripting capability.

  • REAPI compatibility: Buck2 supports the same Remote Execution API as Bazel, enabling it to use existing RBE infrastructure (BuildBuddy, EngFlow, etc.).

  • Hermetic execution: Same sandbox model as Bazel - only declared inputs accessible to actions.

Origin: Buck (v1) was created by Meta (then Facebook) in 2012. Buck2 was built from scratch starting circa 2020 and open-sourced in

Think of Buck2 as Bazel redesigned in Rust with one key innovation: every computation is memoized, including the computation of the build graph itself. In Bazel, if you change a BUILD file, Bazel re-analyzes that package. In Buck2, only the specific DICE nodes that depend on the changed value are re-computed; everything else is instantly cached.

Persona pitches:

Junior developer: “Buck2 is a fast build tool that understands your project’s dependency structure and only rebuilds what actually changed. It’s like Bazel but faster, especially for large monorepos.”

Senior engineer: “Buck2 replaces Bazel’s three-phase build model (loading/analysis/execution) with a unified DICE-based incremental computation graph. Every function call in the build system is a memoized DICE node; invalidation is fine-grained to the specific value that changed, not the entire package.”

Staff architect: “Buck2 is a step toward a unified incremental build and development environment - the DICE framework could in theory compute any developer tool artifact (docs, IDE indexes, lint results) with the same fine-grained incrementality as build actions. The architectural bet is that a single memoization infrastructure beats multiple siloed tools.”

🔩 First Principles Explanation

ESSENTIAL COMPLEXITY:

A build system must answer: “given these sources, what artifacts need to be (re)built?” This is fundamentally a change detection and dependency computation problem. The quality of the answer (what to rebuild) directly determines build performance.

NAIVE APPROACH: Rebuild everything on every run. Correct but slow.

MAKEFILE APPROACH: Timestamp comparison. Rebuild if source is newer than target. Fast but wrong (doesn’t detect content changes, doesn’t handle transitive deps).

BAZEL APPROACH: Content-addressed action digest. Rebuild if action inputs change (by hash). Correct and fast, but re-runs full analysis phase on BUILD file changes.

BUCK2 APPROACH (DICE):

DICE models the build graph as a dependency graph of memoized computations. Each computation is a function f(inputs) → output. DICE tracks:

  • The value of each computation
  • Which computations depend on which other computations
  • Which input values changed since last run

When an input changes, DICE marks only the nodes that transitively depend on that input as “potentially dirty.” Re-running a dirty node re-computes its function; if the result is the same as the cached result, downstream nodes remain clean (this is called “check propagation” - a node is dirty only if its VALUE changed, not just if an upstream input was re-evaluated).

Example:

Source.java → compile → Source.class → test → TestResult

If a comment changes in Source.java:

  • compile is re-run (Source.java is dirty)
  • The resulting Source.class is IDENTICAL (comments don’t affect bytecode)
  • DICE detects Source.class didn’t change
  • test node is NOT re-run (its input, Source.class, is unchanged)
  • Test result served from cache instantly

In Bazel, this “early cutoff” optimization also exists but is not as deeply integrated into every part of the build system. In Buck2, DICE enforces it consistently across all computation types.

ACCIDENTAL COMPLEXITY:

  • Buck2 bxl scripting adds a new language surface
  • DICE implementation is complex (concurrent, distributed, GC-free)
  • Rule porting from Bazel (not 100% compatible API)
  • Smaller ecosystem than Bazel (fewer pre-built rules)

🧪 Thought Experiment

The Spreadsheet Model:

Imagine your build graph as a spreadsheet. Each cell contains a value (a build artifact, a test result, a dependency list). Each cell has a formula (a build rule or action). When you change a cell (modify a source file), only the cells whose formulas reference that cell (directly or transitively) need to recalculate.

Excel implements this well. You don’t recalculate the entire spreadsheet when you change one number - you recalculate only the cells that depend on it. And crucially: if recalculating cell B produces the same value as before, cells that depend on B don’t recalculate (this is the “early cutoff” optimization).

Buck2’s DICE is exactly this model applied to build graphs. It’s a reactive, incremental computation framework where:

  • Cells = DICE nodes (source files, build targets, test results)
  • Formulas = build rules and actions
  • Cell dependencies = build target dependencies
  • Excel’s dirty-bit mechanism = DICE’s invalidation propagation

What breaks the analogy: Unlike Excel, DICE runs in parallel (multiple “cells” recalculate simultaneously on a multi-core machine or across remote workers). Also, “cells” in Buck2 can have complex provenance - a test result depends not just on the test binary but on the specific input data provided to the test.

What changes at 10x scale? At 10x engineers (1000), DICE’s ability to share computation state across the daemon process means N developers querying the same build graph simultaneously get cache hits for unchanged nodes. The daemon becomes a shared computation server, not just per-developer state. At 100x (Meta scale), the daemon model would need distribution (DICE was designed for this). At 1000x, the build system itself becomes a distributed database of memoized computations.

🧠 Mental Model / Analogy

Buck2 is to builds what React is to UI. React introduced the concept of a virtual DOM - a declarative representation of what the UI should look like - and a reconciliation algorithm that computes the minimal set of DOM changes needed. Buck2 introduces a declarative build graph (Starlark rules) and a reconciliation algorithm (DICE) that computes the minimal set of build actions to re-execute. Both systems use the insight that re-computing from scratch is often simpler to reason about than tracking imperative mutations, but the smart incremental execution engine handles performance.

A shorter version: Buck2 is a build system whose internal state machine always asks “what changed?” before asking “what to do?” This is a push model vs pull model distinction - instead of Bazel pulling the full analysis graph on every build, Buck2 is notified of specific changes and propagates invalidation lazily.

📶 Gradual Depth - Five Levels

Level 1 - User: Run buck2 build //... to build your project. It’s fast because it remembers what it built before and only rebuilds what changed. Configure targets in BUCK files (very similar to Bazel BUILD files).

Level 2 - Developer: Buck2 uses the same Starlark BUILD files as Bazel. Your rules declare srcs, deps, and output types. Buck2 builds a dependency graph, runs only actions whose inputs changed, and caches results. Use buck2 query to explore dependencies; buck2 test to run tests.

Level 3 - Engineer: DICE (Distributed Incremental Computation Engine) tracks all build computations as memoized function calls. On each build, DICE re- evaluates only nodes whose inputs changed - including BUILD files. The “early cutoff” optimization means a node whose output value is unchanged does not propagate invalidation to its consumers. This is why changing a comment in Java doesn’t re-run tests.

Level 4 - Senior: Buck2 rules are written in Starlark. The rule API differs from Bazel: ctx.actions.run() is familiar but provider creation is slightly different. bxl scripts extend the system with programmatic graph queries. Buck2 supports --oncall for build target ownership and integrates with Meta’s internal error classification system. REAPI remote execution is configured via remote_execution config in .buckconfig.

Level 5 - Architect: DICE is a demand-driven, incremental, parallel computation framework. It solves a fundamental problem in build systems: the cost of re-running analysis when the build graph is large. By treating BUILD file evaluation, rule analysis, and action execution as one consistent memoized graph, Buck2 eliminates the “cold analysis” phase that Bazel must run after BUILD file changes. The daemon holds all of DICE state in memory; this trades memory for latency. At Meta’s scale, the daemon is a critical path dependency - it must be highly available and support concurrent multi-user access. The bxl scripting layer enables codegen workflows and custom analysis passes to be co-located with the build graph computation, avoiding separate tools that drift out of sync.

⚙️ How It Works (Mechanism)

Buck2 Build Phases:

buck2 build //my:target

1. LOAD PHASE (DICE-backed)
   ├── Read BUCK file for //my package
   ├── Execute Starlark to produce target nodes
   ├── DICE caches result (same BUCK file = instant)
   └── Transitively load all dependencies

2. ANALYSIS PHASE (DICE-backed)
   ├── Run each rule's impl function
   ├── Collect providers from deps
   ├── Create action graph (ctx.actions.run, etc.)
   └── DICE caches analysis per (rule, attrs, dep_providers)

3. EXECUTION PHASE (action execution)
   ├── Check remote Action Cache
   ├── Execute cache-miss actions (local or remote)
   ├── Materialise outputs (download from CAS)
   └── DICE caches execution result

4. QUERY PHASE (bxl or buck2 query)
   ├── Re-uses DICE state from build
   ├── Instant re-query if nothing changed
   └── Returns from DICE cache without reanalysis

Code walkthrough: The key difference from Bazel is phases 1-3 are all DICE nodes, not separate phases with separate invalidation logic. Changing a .bzl rule file in Bazel invalidates all targets that use it (full re-analysis). In Buck2, DICE tracks exactly which analysis nodes depend on that .bzl file and re-evaluates only those. Phase 4 (query) reusing phase 1-3 DICE state means buck2 query "deps(//...) after a build returns immediately - it’s reading from the already-computed DICE cache.

bxl Example:

# query.bxl: Find all Java targets with no tests
def _find_untested(ctx):
    # Query all Java targets:
    java_targets = ctx.configured_targets(
        "kind(java_library, //...)"
    )
    untested = []
    for tgt in java_targets:
        # Check if there's a test target for this:
        test_pattern = tgt.label.package + ":*_test"
        tests = ctx.configured_targets(test_pattern)
        if not tests:
            untested.append(str(tgt.label))
    ctx.output.print("\n".join(untested))

untested_java = bxl_main(
    impl = _find_untested,
    cli_args = {},
)

# Run the bxl script:
buck2 bxl //tools:query.bxl:untested_java

Code walkthrough: This bxl script uses the DICE-backed build graph to find Java libraries without corresponding test targets. ctx.configured_targets() queries the fully-analyzed build graph (not just the raw BUCK files), meaning it respects select() conditions and actual configured attributes. The result is an actionable list of untested modules - a code quality gate that would be complex to implement with bazel query (which operates on the unconfigured graph). bxl scripts run inside the DICE framework, so re-running the same query returns instantly from cache.

Example 1 - Buck2 BUCK file vs Bazel BUILD file

# BAD (Bazel BUILD syntax in a Buck2 project):
# Using Bazel-specific built-ins that don't exist in Buck2
java_library(
    name = "payments",
    srcs = glob(["src/main/java/**/*.java"]),
    deps = ["@maven//:com_fasterxml_jackson_core"],
    # WRONG: @maven// is Bazel's Maven integration
    # Buck2 uses different dependency declaration
)

# GOOD (Buck2 BUCK file):
java_library(
    name = "payments",
    srcs = glob(["src/main/java/**/*.java"]),
    deps = [
        # Buck2 uses prebuilt_jar or maven_jar:
        "//third-party/java:jackson-databind",
    ],
    # Buck2 visibility declaration:
    visibility = ["PUBLIC"],
)

# Third-party definition (in //third-party/java/BUCK):
prebuilt_jar(
    name = "jackson-databind",
    binary_jar = "jackson-databind-2.15.1.jar",
    visibility = ["PUBLIC"],
)

Code walkthrough: The key structural difference between Bazel and Buck2 third-party dependency management. Bazel uses repository rules (@maven//) which fetch from Maven Central during the WORKSPACE/module phase. Buck2 typically uses pre-downloaded JARs referenced via prebuilt_jar in a third-party directory. Both approaches work; the Buck2 approach is more hermetic by default (no network access at build time). The visibility = ["PUBLIC"] in Buck2 defaults to package-private if omitted, whereas Bazel requires explicit visibility settings.

Example 2 - bxl for build graph analysis (production use)

# tools/find_heavy_deps.bxl
# Find targets with transitive dep count > threshold.
# Production use: identify targets that cause large
# rebuilds when changed.

def _find_heavy(ctx):
    threshold = ctx.cli_args.threshold
    all_targets = ctx.configured_targets("//...")
    heavy = []
    for tgt in all_targets:
        # Count transitive deps using DICE graph:
        deps = ctx.configured_targets(
            "deps(%s)" % str(tgt.label)
        )
        count = len(list(deps))
        if count > threshold:
            heavy.append(
                "%s: %d deps" % (tgt.label, count)
            )
    # Sort by dep count (highest first):
    heavy.sort(key=lambda x: -int(x.split(": ")[1]
        .split(" ")[0]))
    for line in heavy[:20]:  # top 20
        ctx.output.print(line)

find_heavy_deps = bxl_main(
    impl = _find_heavy,
    cli_args = {
        "threshold": cli_args.int(
            default = 100,
            doc = "Min transitive dep count"
        ),
    },
)

Code walkthrough: This bxl script implements a build graph health check: finding targets with large transitive dependency footprints. A target with 500 transitive deps is a “heavy target”

  • changing it or any of its deps triggers a large rebuild cascade. The script uses ctx.configured_targets("deps(%s)") which is equivalent to bazel query "deps(//target)" but runs inside DICE, meaning re-running it returns instantly from cache. Production use: run this as a CI gate and flag targets whose dep count grows beyond threshold, preventing accidental “spaghetti dependency” growth.

Example 3 - Migrating a Bazel rule to Buck2

# Bazel rule (rules/proto/BUILD):
def proto_java_library(name, proto_srcs, **kwargs):
    # Bazel: use native.java_proto_library
    native.proto_library(
        name = name + "_proto",
        srcs = proto_srcs,
    )
    native.java_proto_library(
        name = name,
        deps = [":" + name + "_proto"],
        **kwargs
    )

# Buck2 equivalent (rules/proto.bzl):
def proto_java_library(name, proto_srcs, **kwargs):
    # Buck2: use prebuilt_jar for protoc output
    # Buck2 macro runs protoc as a genrule:
    genrule(
        name = name + "_gen",
        srcs = proto_srcs,
        cmd = ("$(exe //tools:protoc) "
               "--java_out=$OUT $SRCS"),
        out = name + "_gen.srcjar",
    )
    java_library(
        name = name,
        srcs = [":" + name + "_gen"],
        deps = kwargs.get("deps", []) + [
            "//third-party/java:protobuf-java",
        ],
        visibility = kwargs.get("visibility", []),
    )

Code walkthrough: The Bazel rule uses native.java_proto_library (a Bazel built-in); Buck2 has no direct equivalent so the macro uses a genrule calling protoc directly. The $(exe //tools:protoc) syntax in Buck2’s genrule declares the protoc binary as a tool dependency (similar to Bazel’s tools attribute). The key porting challenge is that Bazel has many more built-in rules than Buck2; Buck2 encourages building more from primitives. This is more verbose but also more explicit about what is happening.

🔁 Flow / Lifecycle

Buck2 Build Lifecycle:

1. DAEMON START (once per session or on config change)
   └── Load initial configuration (.buckconfig)
   └── Initialize DICE state

2. COMMAND RECEIVED: buck2 build //target
   ├── DICE: load all transitive BUCK files
   │   (cache hits for unchanged packages)
   ├── DICE: analyze all target rules
   │   (cache hits for unchanged rules + deps)
   ├── DICE: create action graph
   └── Schedule actions for execution

3. EXECUTION
   ├── Check remote Action Cache (REAPI)
   ├── Execute cache-miss actions
   └── DICE: update execution results + early cutoff

4. OUTPUT
   └── Materialize outputs to buck-out/

5. INCREMENTAL BUILD (repeat with source change)
   ├── DICE detects changed source file
   ├── Propagate invalidation to dependent nodes only
   ├── Re-execute only dirty nodes
   └── Early cutoff if value unchanged

⚠️ Common Misconceptions

1. “Buck2 and Bazel are fully compatible” WRONG. They use the same Starlark language but different built-in rule APIs. java_library exists in both but with different available attributes. @maven// doesn’t exist in Buck2. Rules written for Bazel need porting to Buck2 (usually minor but not zero effort).

2. “Buck2 is faster because it skips analysis” PARTIALLY WRONG. Buck2 is faster because it avoids redundant analysis via DICE memoization - it doesn’t skip analysis for changed nodes. What it avoids is re-analyzing nodes that were NOT affected by a change, which Bazel’s analysis phase doesn’t always avoid.

3. “DICE means Buck2 never needs to restart its daemon” WRONG. DICE state is invalidated when .buckconfig changes, when build files in the root are modified, or when configuration flags change. Daemon restart clears all DICE state (cold start). Buck2’s buck2 kill and buck2 clean commands clear state explicitly.

4. “bxl replaces buck2 query” WRONG. buck2 query is a simple query interface for the unconfigured build graph. bxl is for complex, scripted queries on the configured graph that need programmatic logic (loops, conditionals, custom output formats). For simple “what are the deps of X?” queries, buck2 query is simpler.

5. “Buck2’s Rust implementation makes it maintenance-free” WRONG. Rust prevents memory safety bugs but not logic bugs. Buck2 requires build engineers to maintain custom rules (in Starlark), keep third-party JARs updated, and tune DICE memory pressure in large monorepos. The Rust implementation reduces operational bugs but doesn’t eliminate maintenance.

🚨 Failure Modes & Diagnosis

FAILURE 1: DICE state corruption

Symptom: Buck2 reports a target as built but the output
  file doesn't exist. Or: incorrect outputs from a cached
  result that should have been invalidated.

Diagnosis:
  buck2 build //target --no-cache
  # Forces full rebuild without DICE cache

  buck2 clean  # Nuclear option: clear all DICE state
  # Equivalent to Bazel's bazel clean --expunge

Root cause:
  Rare DICE state inconsistency (usually after daemon
  crash during write). Also: stale buck-out state from
  a previous version of Buck2.

Fix:
  buck2 clean && buck2 build //target
  # Clear DICE state and rebuild

FAILURE 2: Rule porting error - missing provider

Symptom: "Error: deps attribute received a target that
  does not provide JavaInfo" (or similar provider error)
  during analysis phase.

Diagnosis:
  buck2 build //failing:target -v 5
  # Verbose output shows which rule and which provider
  buck2 query "kind('java_library', deps(//failing))"
  # Check which deps are java targets

Root cause:
  A dep target doesn't return the expected provider.
  Common during Bazel-to-Buck2 migration: Bazel rule
  ported but provider return differs.

Fix:
  Add missing provider to the dep rule's impl function.
  Or: change the consumer rule to handle missing provider.

FAILURE 3: Remote execution auth failure

Symptom: "authentication failed" or "permission denied"
  in Buck2 RBE output. Falls back to local execution.

Diagnosis:
  # Check token:
  echo $BUCK2_RBE_TOKEN | cut -c1-20  # Verify non-empty
  # Check expiry (if JWT):
  echo $BUCK2_RBE_TOKEN | cut -d. -f2 | base64 -d 2>/dev/null

Root cause:
  Service account token expired, rotated, or not set.

Fix:
  Refresh token: regenerate SA key or re-authenticate.
  Set in environment before running buck2.

FAILURE 4: bxl script memory pressure

Symptom: bxl script OOMs the Buck2 daemon.
  Error: "Out of memory during bxl execution"

Diagnosis:
  # Count query result size:
  buck2 query "deps(//...)" --count
  # If count > 100,000: bxl materializing all nodes
  # in memory is expensive

Root cause:
  bxl script calls ctx.configured_targets("deps(//...)")
  and iterates over 500,000+ nodes materializing all.

Fix:
  Scope the query: ctx.configured_targets("deps(//my_pkg)")
  Or: use streaming output instead of collecting to list.
  Or: add depth limit: "deps(//..., 3)" (3 levels max).

💎 Transferable Wisdom

DICE’s early cutoff is a general optimization pattern: The insight that “if the output didn’t change, don’t propagate the invalidation” appears in many reactive systems: React’s React.memo, Redux selectors (reselect), computed properties in Vue/MobX, and database query result caching with TTL-bypass on unchanged rows. When building any system that recomputes on input changes, always ask: can I use value equality to short- circuit downstream recomputation?

Daemon architectures trade memory for latency: Buck2’s DICE daemon holds all build graph state in memory (like Bazel’s analysis server). This trades RAM for build latency. The same pattern appears in language servers (LSP), test watchers (Jest –watch), and hot-reloading dev servers (Vite). The tradeoff is correct when interactive latency is more valuable than memory. Know when to restart the daemon (state gets too stale or corrupt) vs trust it.

Tool competition drives innovation: Bazel’s competition with Buck2 (and Pants, Gradle, etc.) has driven improvements in all three tools. Bazel adopted analysis caching improvements partially in response to Buck2’s DICE model. This is a pattern in platform engineering: multiple competing implementations of a standard (REAPI, Starlark) improve each other faster than a monopoly would. When evaluating build tools, understand which capabilities are unique to each and which are converging.

💡 The Surprising Truth

Buck2’s biggest performance win is not in the build itself - it’s in the query.

Teams often evaluate build systems by clean-build time. But for large monorepos, the majority of developer time is in incremental builds AND tool queries: “what are the deps of X?”, “which targets are affected by this change?”, “which tests do I need to run?”

In Bazel, bazel query runs a fresh analysis of the queried subgraph every time. For a deeply nested dependency query, this can take 10-30 seconds. In Buck2, buck2 query after a recent build returns in milliseconds - it reads from DICE’s already- computed analysis cache.

For developer experience, the latency of “I changed a file, what do I need to re-run?” is often more impactful than clean build time. Buck2’s DICE architecture optimizes exactly this use case. The surprising conclusion: the best build system optimization for developer velocity is often making the build graph queryable in real time, not making the compilation faster.

🏗️ Model Answer Framework

WHAT is Buck2? Buck2 is Meta’s open-source Rust-based build system that uses Starlark BUILD files and the DICE (Distributed Incremental Computation Engine) framework to provide fine-grained incremental builds with sub-second query latency and REAPI-compatible remote execution.

WHY does it exist? Buck1 (JVM-based) had GC pauses and insufficient incrementality for Meta’s monorepo scale. The core insight: treating every build system computation (not just actions) as a memoized function enables invalidating only exactly what changed - including BUILD file analysis - rather than re-running full analysis passes.

HOW IT WORKS: DICE creates a graph of memoized computations: BUCK file loading, rule analysis, action execution, and test results. When a source file changes, DICE invalidates only the nodes that transitively depend on it. Early cutoff prevents downstream invalidation when a node’s output value is unchanged despite an upstream change.

TRADE-OFF (gain vs cost):

  • GAIN: Sub-second incremental build queries; faster interactive iteration; same REAPI/Starlark ecosystem as Bazel
  • COST: Smaller rule ecosystem than Bazel; porting effort from Bazel/Maven; DICE daemon memory footprint; bxl adds new language surface to maintain

EXAMPLE (production): Meta’s monorepo (hundreds of millions of lines) uses Buck2 in production. The killer feature for Meta: changing a single Java file and running buck2 test :... returns affected test results in seconds rather than minutes, because DICE avoids re-analyzing the thousands of unchanged packages.

🎯 Interview Deep-Dive

Q1: What is DICE and why does it matter for build performance?

DICE is Buck2’s Distributed Incremental Computation Engine. It models every computation in the build system - BUILD file loading, rule analysis, action execution - as a memoized function f(inputs) → output. DICE tracks which computations depend on which others.

When an input changes, DICE marks dependent nodes as “potentially dirty.” On the next build, dirty nodes re-compute their function. The key innovation is “early cutoff”: if re-computing a node produces the same output value as before (e.g., a Java source change that doesn’t affect compiled bytecode), DICE does NOT propagate the dirty mark to downstream consumers. Tests that depend on the unchanged .class file are not re-run.

This matters because it unifies what Bazel separates (analysis phase vs execution phase) into one consistent incremental model. BUILD file analysis is memoized just like action execution, making queries and incremental builds fast.

Q2: How does Buck2 handle third-party dependencies differently from Bazel?

Bazel uses repository rules (e.g., rules_jvm_external) that download JARs from Maven Central during the loading phase (WORKSPACE or MODULE.bazel). This requires network access during builds and introduces a fetching step.

Buck2 typically uses prebuilt_jar targets in a third-party directory: JARs are checked into the repo (or downloaded separately and placed in a directory). Build time requires no network access. This is more hermetic but requires a separate dependency management workflow (e.g., a script that downloads JARs to the third-party directory and generates BUCK files).

Neither approach is strictly better: Bazel’s bzlmod provides better version conflict resolution; Buck2’s prebuilt approach is simpler and fully offline.

Q3: When would you choose Buck2 over Bazel for a new project?

Choose Buck2 over Bazel when:

  1. The team is primarily building Meta-originated infrastructure (Buck2 has better integration with fbthrift, folly, etc.)
  2. Interactive build latency is the primary developer pain point (DICE provides faster queries and incremental rebuilds)
  3. Rust expertise is available on the team (debugging Buck2 issues requires Rust knowledge for the core; Bazel requires Java)
  4. The monorepo is growing toward Meta-scale where Bazel’s analysis phase costs become significant

Choose Bazel when:

  1. Larger ecosystem of existing rules is needed (rules_go, rules_k8s)
  2. The team has existing Bazel expertise
  3. Better community documentation and support is needed
  4. Integration with Google Cloud or specific GCP tooling is needed

Q4: What is bxl and when would you use it over buck2 query?

buck2 query operates on the unconfigured build graph (without applying select() conditions or configuration transitions). It’s suitable for simple dependency queries: “what are the deps of //X?”

bxl (Buck2 Extension Language) operates on the configured build graph inside the DICE computation framework. It’s a Starlark scripting API that provides: programmatic logic (loops, conditionals), access to configured attributes and providers, and the ability to run actions as part of the analysis.

Use bxl when:

  • The query requires conditional logic
  • You need to inspect providers (what rules return to their consumers)
  • You’re building a code generation workflow that depends on build graph analysis
  • You want the query to be memoized and re-usable without re-analysis

Q5: Explain Buck2’s approach to remote execution. How does it compare to Bazel’s?

Buck2 uses the same REAPI standard as Bazel for remote execution. The configuration differs: Buck2 uses .buckconfig sections ([buck2_re_client]) while Bazel uses --remote_executor flags or .bazelrc. Both connect to the same RBE servers (BuildBuddy, EngFlow, NativeLink, Google RBE).

The key architectural difference: Buck2’s DICE framework means the action execution results are integrated into the same DICE graph as analysis results. When a remote action completes, DICE updates the execution node, applies early cutoff checking, and only propagates to downstream nodes if the output changed. In Bazel, the remote execution result is returned to the action graph but analysis cache is separate; there is no equivalent of early cutoff for the analysis layer.

In practice: for pure remote execution performance, the systems are comparable (same REAPI calls, same CAS/AC protocol). Buck2’s advantage is in the invalidation propagation after execution.

🌍 Sustainability & Ethics

ENERGY EFFICIENCY VIA EARLY CUTOFF: Buck2’s early cutoff optimization has direct energy implications: if a comment change to a Java file triggers recompilation but the resulting .class is identical, zero test actions are re-executed. Without early cutoff, the same comment change would invalidate and re-run all tests that transitively depend on that class. At a 1000-engineer monorepo with thousands of tests, the difference is millions of CPU-seconds avoided per day.

RUST’S LOWER ENERGY FOOTPRINT: Buck2’s Rust implementation has a lower memory and CPU footprint than Bazel’s JVM implementation for equivalent operations. No JVM heap means no GC pauses and lower steady-state memory consumption. For long-running daemons on developer machines, this translates to less battery drain and fan activity.

OPEN-SOURCE COMMITMENT: Meta open-sourced Buck2 in 2023 under Apache-2.0, making its innovations (DICE, bxl) available to the broader engineering community. This prevents vendor lock-in and allows smaller organizations to benefit from Meta’s build engineering investment. The open-source gesture also acknowledges that build tooling is a public good rather than a competitive differentiator.

buck2 kill

buck2 clean

buck2 build //payments:server

# Check for daemon crash during previous build:
